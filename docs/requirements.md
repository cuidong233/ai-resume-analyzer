# AI 赋能的智能简历分析系统需求文档

## 1. 项目背景

招聘流程中，HR 或招聘负责人需要在短时间内筛选大量简历。人工阅读 PDF 简历、提取候选人关键信息、再与岗位 JD 逐项对照，效率低且容易遗漏。项目目标是构建一个可部署到云端的简历分析系统，支持上传 PDF 简历，自动解析文本，提取结构化信息，并根据岗位需求计算匹配度评分，辅助招聘者完成候选人初筛。

## 2. 项目目标

- 提供 RESTful API，支持单个 PDF 简历上传与解析。
- 兼容多页 PDF 简历，完成文本提取、清洗和分段。
- 使用 AI 模型或规则兜底方式提取候选人关键信息。
- 接收岗位需求描述，提取岗位关键词并计算简历匹配度。
- 以 JSON 格式返回解析结果、结构化信息、评分和匹配原因。
- 提供简洁可用的前端页面，支持在线演示。
- 后端可部署到阿里云 Serverless 函数计算 FC。

## 3. 用户角色

- 招聘者：上传候选人简历，输入岗位需求，查看匹配结果。
- 评审人员：通过线上页面或 API 文档验证系统功能完整性、代码质量和部署可行性。

## 4. 功能需求

### 4.1 简历上传与解析

必选需求：

- 支持上传单个 PDF 格式简历文件。
- 校验文件类型、空文件和文件大小。
- 解析 PDF 文本内容，支持多页简历。
- 对文本进行清洗，去除冗余空白和异常字符。
- 对清洗后的文本进行合理分段，便于后续抽取和展示。

异常处理：

- 非 PDF 文件返回 400。
- 空文件返回 400。
- 超过大小限制返回 413。
- 无法提取文本的扫描件返回明确错误提示。

### 4.2 关键信息提取

必选字段：

- 姓名
- 电话
- 邮箱
- 地址

加分字段：

- 求职意向
- 期望薪资
- 工作年限
- 学历背景
- 技能列表
- 项目经历
- 简历摘要

实现要求：

- 优先通过 AI 模型进行结构化抽取。
- 未配置 AI Key 或 AI 调用失败时，使用规则抽取作为兜底方案。
- 不编造简历中不存在的信息，缺失字段返回 `null` 或空数组。

### 4.3 简历评分与岗位匹配

必选需求：

- 提供接口接收岗位需求描述。
- 从岗位需求中提取关键词。
- 将候选人简历与岗位需求进行匹配。
- 输出综合匹配度评分。
- 输出技能匹配率、工作经验相关性、学历匹配度、关键词覆盖率等维度评分。
- 返回命中关键词、缺失关键词和匹配说明。

加分实现：

- 接入 AI 模型复核匹配度评分，使评分更贴近真实招聘判断。
- AI 不可用时，使用本地规则评分保证系统可演示。

### 4.4 结果返回与缓存

必选需求：

- 所有接口返回 JSON。
- 解析结果包含 `resume_id`、原始清洗文本、分段、结构化档案和缓存命中状态。
- 匹配结果包含综合分、分项分、岗位关键词、命中/缺失关键词、AI 说明和缓存命中状态。

加分实现：

- 支持 Redis 缓存。
- 未配置 Redis 时使用内存缓存。
- 对相同简历解析结果和相同岗位匹配结果进行缓存，避免重复计算。

### 4.5 前端页面

必选需求：

- 提供可公开访问的前端页面。
- 支持上传 PDF 简历。
- 支持粘贴文本用于快速演示。
- 支持输入岗位 JD。
- 展示解析结果、关键信息、匹配评分和关键词命中情况。

部署要求：

- 前端可部署到 GitHub Pages、Vercel、Netlify 或其他公开云服务。
- 前端通过环境变量配置后端 API 地址。

## 5. 非功能需求

### 5.1 运行环境

- 后端语言：Python。
- 后端框架：FastAPI。
- 部署平台：阿里云 Serverless 函数计算 FC。
- 前端框架：React + Vite。
- 缓存：Redis，可选；内存缓存兜底。

### 5.2 性能要求

- 常规文本型 PDF 简历应在数秒内完成解析。
- AI 调用设置超时，避免请求长时间阻塞。
- 通过缓存减少重复解析和重复评分。

### 5.3 可维护性要求

- 后端按 API、模型、服务层拆分。
- PDF 解析、AI 调用、规则抽取、缓存、匹配评分相互独立。
- README 说明架构、部署方式、环境变量和使用方式。
- 提供基础测试覆盖规则抽取和匹配逻辑。

### 5.4 安全与隐私要求

- 限制上传文件大小。
- 不在仓库提交真实 API Key。
- 通过 `.env.example` 提供环境变量示例。
- 生产环境应配置 CORS 白名单。
- 简历数据如需长期存储，应增加脱敏、访问控制和数据清理策略。

## 6. API 需求

### 6.1 健康检查

`GET /health`

返回：

```json
{
  "status": "ok",
  "ai_enabled": true,
  "redis_enabled": false
}
```

### 6.2 上传并解析简历

`POST /api/resumes`

请求类型：`multipart/form-data`

字段：

- `file`: PDF 简历文件

返回核心字段：

- `resume_id`
- `file_name`
- `text`
- `sections`
- `profile`
- `cache_hit`

### 6.3 文本解析

`POST /api/resumes/text`

用途：便于演示或调试，直接粘贴简历文本完成解析。

请求示例：

```json
{
  "text": "候选人简历文本"
}
```

### 6.4 岗位匹配评分

`POST /api/match`

请求示例：

```json
{
  "resume_id": "已解析简历 ID",
  "job_description": "岗位需求文本"
}
```

也支持直接传入简历文本：

```json
{
  "resume_text": "候选人简历文本",
  "job_description": "岗位需求文本"
}
```

返回核心字段：

- `score`
- `breakdown.skill_match`
- `breakdown.experience_relevance`
- `breakdown.education_match`
- `breakdown.keyword_coverage`
- `job_keywords`
- `matched_keywords`
- `missing_keywords`
- `ai_comment`
- `cache_hit`

## 7. 技术方案

### 7.1 后端方案

- 使用 FastAPI 提供 RESTful API。
- 使用 `pypdf` 解析文本型 PDF。
- 使用 Pydantic 定义结构化返回模型。
- 使用 OpenAI 兼容 Chat Completions 接口进行 AI 抽取和评分复核。
- 使用规则抽取作为离线兜底，保证无 Key 环境也可运行。
- 使用 Redis 或内存缓存保存解析和匹配结果。
- 使用 Serverless Devs `s.yaml` 提供阿里云 FC 部署示例。

### 7.2 前端方案

- 使用 React + Vite 构建单页应用。
- 支持 PDF 上传、文本演示、JD 输入和匹配评分展示。
- 使用环境变量 `VITE_API_BASE_URL` 指向后端服务。
- 静态产物可部署到 GitHub Pages 或其他静态托管平台。

## 8. 验收标准

- 本地后端可启动，`/health` 返回正常。
- 上传文本型 PDF 后，可返回简历文本、分段和结构化信息。
- 输入岗位 JD 后，可返回匹配评分、分项评分和关键词分析。
- 未配置 AI Key 时，系统仍可通过规则抽取和本地评分完成演示。
- 前端页面可完成上传、解析、匹配和结果展示流程。
- README 包含项目架构、技术选型、部署方式和使用说明。
